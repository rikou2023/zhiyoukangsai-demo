## 3. P1 问题

### P1-1. 页面存在横向溢出

| 视口 | 实测结果 |
|---|---|
| 桌面端 | `scrollWidth 1616 > clientWidth 1425` |
| 移动端首屏 | `scrollWidth 398 > clientWidth 375` |
| 移动端菜单打开后 | `scrollWidth 417 > clientWidth 390` |

浏览器检测到的主要溢出来源：

| 来源 | 证据 |
|---|---|
| Hero AI 平台跑马灯 | `.marquee-track` 宽度远超视口，多个 `.platform-logo` 出现在视口外 |
| 系统演示轮播 | `.showcase-slide` / 截图元素在桌面检测中出现右侧超出 |
| Canvas / 全屏背景 | `.bg-neural` 宽度贴近临界，移动端与滚动条宽度叠加后容易触发溢出 |

相关位置：

| 文件 | 位置 |
|---|---|
| `website/index.html` | `platform-marquee` 在 111 行附近，`showcase-slide` 在 158 行附近 |
| `website/assets/css/index.css` | `.marquee-track` 549 行附近，`.marquee-viewport` 557 行附近，`.showcase-slide` 752 行附近 |
| `website/assets/css/tokens.css` | `body { overflow-x: hidden; }` 在 125 行附近 |

影响：

- 这是显示层面的硬伤，即使 `body` 隐藏了横向滚动，也说明有元素超出布局边界。
- 移动端可能出现轻微左右晃动、截图裁切、点击热区异常。
- 后续如果换浏览器或嵌入第三方环境，风险会放大。

建议：

1. 给所有跑马灯外层加更明确的 `max-width: 100vw; overflow: hidden;`。
2. 避免动画轨道影响文档流宽度，可让 `.marquee-track` 使用绝对定位或在已裁切容器内运行。
3. 系统演示轮播的离场 slide 不应参与页面宽度计算，确认父级 `overflow:hidden` 和定位上下文是否完整。
4. 不要只依赖 `body { overflow-x:hidden; }` 掩盖问题，要让 `documentElement.scrollWidth` 回到等于 `clientWidth`。

### P1-2. 固定头部遮挡滚动后的区块标题

桌面端从首屏滚到痛点段时，看到“客户在问 Kimi、文心一言、通义千问...”大标题顶部被固定导航压住，形成明显裁切。

相关位置：

| 文件 | 位置 |
|---|---|
| `website/assets/css/tokens.css` | `--header-height: 72px` 在 70 行附近 |
| `website/assets/css/index.css` | `.site-header` fixed 在 9 行附近 |

影响：

- 关键内容显示不完整，用户会以为页面有 bug。
- 锚点跳转到 `#contact-form` 或其他区块时，也可能被 header 遮挡。

建议：

1. 给主要 section 增加统一的 `scroll-margin-top: calc(var(--header-height) + 24px);`。
2. 对痛点段这种大标题区，额外增加上方安全距离。
3. 验证所有 CTA 锚点跳转后的目标标题/表单首行不被头部遮挡。

---

### P1-3. 数字动画有暴露初始 0 值的风险

源码中多处数字初始文案为 `0`、`0%`、`0亿+`，依赖 IntersectionObserver 和 countup 动画更新。

相关位置：

| 文件 | 位置 |
|---|---|
| `website/index.html` | 案例数字：500、551、587 行附近 |
| `website/index.html` | 资质数字：631、635、639、643 行附近 |
| `website/assets/js/index.js` | 通用 countup：653-691 行附近 |
| `website/assets/js/index.js` | 案例切换时 reset 到 0：1006-1012 行附近 |

影响：

- 测试文本流里可以看到 `0%`、`0 国家级核心技术专利`、`0 软件著作权` 等内容。
- 如果动画未触发、滚动截图太快、浏览器禁用动画、脚本异常，客户会看到不可信数字。

建议：

1. HTML 初始值直接写最终值，例如 `300%`、`1.1亿+`、`37.22%`、`4`、`6`、`100+`、`30+`。
2. countup 动画只做视觉过渡，不把真实业务数字依赖在 JS 上。
3. 案例轮播切换时不要先裸露 `0%`，可用淡出旧数字、淡入新数字，或在不可见期间重置。

## 4. P2 问题

### P2-1. 首屏有科技感，但业务视觉还可以更强

现状：

- 深色背景、神经网络、AI 平台轮换都符合 GEO 主题。
- 主标题有记忆点，“AI 第一推荐位”表达清楚。
- 但首屏仍偏“标题 + 背景动效 + Logo 跑马灯”，真正表达 GEO 结果的业务画面不够强。

建议：

- 在首屏或首屏下沿加入一个更具体的“AI 推荐位监测”视觉，例如：
  - AI 回答卡片中品牌被推荐；
  - 推荐位排名从竞品切换到客户品牌；
  - DeepSeek / Kimi / 文心一言多平台推荐状态面板；
  - “被 AI 看见 -> 被客户选择”的短流程。

目标不是加装饰，而是让视觉本身解释业务。

---

### P2-2. 深层内容偏普通 SaaS 模板

现状：

- 后半页出现较多“标题 + 说明 + 卡片/列表”的常见结构。
- 解决方案、案例、评价、Logo 墙内容完整，但视觉节奏仍比较平。
- 和 Tec-Do 那类强品牌官网相比，当前页面的视觉高潮主要停留在首屏。

建议：

| 区块 | 建议方向 |
|---|---|
| 痛点段 | 保持大问句，但减少移动端蓝色高亮堆叠，增加 AI 回答截图式对比 |
| 系统演示 | 让系统截图成为真正主角，加入“AI 平台、提示词、引用源、推荐位”的标注层 |
| 案例数据 | 不只放大数字，增加“优化前/优化后”对比或推荐位变化轨迹 |
| 资质背书 | 做成权威徽章墙或时间线，减少普通数字卡片感 |
| 表单区 | 加强专家诊断承诺，说明提交后会获得什么具体报告 |

---

### P2-3. 客户评价可信度不足，有明显占位感

现状：

- 评价文案大量使用“某知名科技公司”“某上市公司”“某企业服务公司”等匿名表达。
- 语气高度一致，容易被判断为 AI 生成或占位内容。
- 数据如 “AI 搜索流量增长 320%”“ROI 1:8.5”“品牌搜索量增长 1200%” 如果没有案例来源，会显得夸张。

建议：

1. 如果不能展示真实公司名，至少使用更可信的行业标签和项目上下文，例如“某跨国营养健康品牌”“某全球头部仪器品牌”。
2. 每条评价控制长度，避免所有评价都像同一套话术。
3. 加入“服务前问题 / 采取动作 / 结果”三段式，让内容更像真实案例。
4. 只保留最强 3-5 条长评，其余做短句滚动。

---

### P2-4. 移动端菜单可用，但视觉偏重

实测：

- hamburger 可点击。
- 菜单打开后 `aria-expanded` 变为 `true`。
- `.mobile-drawer.is-open` 生效。
- `body.drawer-open` 生效。

问题：

- 菜单卡片较厚，所有导航项都像大按钮，层级略重。
- 打开菜单后页面仍可看到下层内容，视觉干扰较强。
- 菜单打开后仍检测到横向溢出。

建议：

1. 增加更强的暗色遮罩或提高抽屉背景纯度。
2. 菜单项可以减轻卡片感，改为更轻的分组列表。
3. 修复横向溢出后复测菜单打开状态。

---

### P2-5. 长页动画和大图较重，影响可测性与性能观感

现象：

- 深滚动截图过程中出现过浏览器截图超时。
- `website/assets/system/` 中 5 张系统截图约 3.1MB-4.1MB/张。
- 首页图片数量较多，动画和轮播也较多。

影响：

- 真实用户在弱网或移动端可能感到加载慢。
- 测试、截图、客户远程预览时可能出现卡顿。

建议：

1. 系统截图转为 WebP，并保留合适尺寸版本。
2. 使用 `srcset` 或移动端低分辨率图。
3. 非首屏图片使用更明确的 lazy loading。
4. 动画区优先支持 `prefers-reduced-motion`，减少移动端动效负担。

## 6. 美感与 AI 风判断

### 6.1 成立的地方

- 深海军蓝 + 电光蓝适合 GEO / AI 搜索服务。
- 神经网络背景和 AI 平台 Logo 不是无意义装饰，和业务有关。
- 首屏标题聚焦，用户能理解“提高品牌在 AI 回答中的推荐位”。
- 页面不是低质 AI 模板，基本具备企业级科技服务官网的方向。

### 6.2 不够高级的地方

- 后半页视觉节奏不够强，缺少一个类似 Tec-Do 那种持续推进叙事的品牌视觉系统。
- 许多区块仍像“组件拼装”：大标题、说明、卡片、数字、Logo 墙。
- 评价和案例内容有明显 AI 文案味，降低真实感。
- 强视觉主要集中在 Hero，后续系统演示和数据区没有形成同等级的视觉记忆点。

### 6.3 新准则下的判断

本页的问题不是“动画太多”或“科技感太强”。相反，当前页可以更大胆。  
真正需要避免的是：统一但机械、科技但空泛、数据多但不可信。

建议后续设计方向：

> 用更强的业务可视化替代泛科技装饰。  
> 让用户看到：品牌如何被 AI 理解、如何被推荐、如何被监测、如何被优化。

## 7. 调整建议优先级

| 优先级 | 调整项 | 目标 |
|---|---|---|
| 1 | 修复桌面和移动端横向溢出 | 消除硬性显示问题 |
| 2 | 修复固定 header 遮挡区块标题 | 解决裁剪/遮盖问题 |

| 6 | 强化系统演示和案例区的业务视觉 | 提升高级感和记忆点 |
| 7 | 微调移动端菜单和痛点段排版 | 提升移动端精致度 |

## 8. 建议修改方案

### 方案 A：先修硬伤，再做视觉增强（推荐）

| 项目 | 内容 |
|---|---|
| 实现思路 | 先解决横向溢出、header 裁切、数字 0 值、大图性能，再增强系统演示和案例区视觉 |
| 影响范围 | CSS、少量 JS、首页内容 |
| 成本 | 中 |
| 风险 | 低，改动集中 |
| 推荐理由 | 最适合当前阶段，先让页面稳定，再提升美感 |

### 方案 B：按强品牌官网重构首页节奏

| 项目 | 内容 |
|---|---|
| 实现思路 | 借鉴 Tec-Do 的叙事强度，重做 Hero、系统演示、案例、表单的视觉节奏 |
| 影响范围 | 首页结构、CSS、动效、部分文案 |
| 成本 | 高 |
| 风险 | 中，可能影响已有内容完整度 |
| 推荐理由 | 适合后续追求更强设计感时做 |

### 方案 C：只做小修小补

| 项目 | 内容 |
|---|---|
| 实现思路 | 只隐藏溢出、调间距、修少量文字 |
| 影响范围 | CSS 为主 |
| 成本 | 低 |
| 风险 | 页面仍会保留模板味 |
| 推荐理由 | 只适合临时预览，不适合最终交付 |

推荐采用 **方案 A**。
