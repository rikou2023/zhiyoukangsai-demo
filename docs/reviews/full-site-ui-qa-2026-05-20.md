## 5. P1 问题

### P1-1. 首页移动端仍存在横向溢出

实测：

| 视口 | 结果 |
|---|---|
| `390 x 844` | `scrollWidth 391 > clientWidth 375` |

主要来源：

- 首页 AI 平台跑马灯 `.marquee-track`
- `.platform-logo`
- preloader 的大 blob 背景元素

相关位置：

| 文件 | 位置 |
|---|---|
| `website/index.html` | 平台跑马灯在 111 行附近 |
| `website/assets/css/index.css` | `.marquee-track` 549 行附近，preloader 在 2923 行附近 |

影响：

- 移动端可能出现轻微左右晃动。
- 截图或客户手机预览时会显得页面没有完全收边。
- 这类问题不能只靠 `body { overflow-x: hidden; }` 掩盖。

建议：

1. 跑马灯外层统一使用 `max-width: 100vw; overflow: hidden;`。
2. 动画轨道不要参与页面宽度计算。
3. preloader 装饰 blob 需要被固定容器裁切。
4. 修完后以 `documentElement.scrollWidth === clientWidth` 作为通过标准。

---

### P1-2. 首页预加载层在快速首屏测试中明显遮挡内容

实测现象：

- 桌面端快速截图时，首屏仍停留在 preloader。
- 移动端等待约 `900ms` 后仍看到 `98%` 加载画面。
- 等待约 `2600ms` 后 preloader 才完全隐藏。

相关位置：

| 文件 | 位置 |
|---|---|
| `website/assets/js/index.js` | `setupPreloader` 在 15 行附近 |
| `website/assets/css/index.css` | `.preloader` 在 2923 行附近 |

影响：

- 本地预览下已经能感到明显等待，弱网或低性能手机上会更重。
- 如果客户只快速打开看首屏，第一印象可能是“还在加载”，而不是看到业务价值。

建议：

1. 首次加载动画压缩到 `800ms-1200ms`。
2. 二次访问可跳过 preloader。
3. 不要把 preloader 当作主视觉，主视觉应该尽快出现。

---

### P1-3. 客户案例数字动画会显示中间值或 0 值

实测现象：

- 桌面端案例页首屏曾看到 `190%`、`0.7亿+`、`23.63%`。
- 移动端案例页曾看到 `297%`、`1.1亿+`、`0%`。
- 这些是动画过程中的中间值，但客户会把它当成真实数据。

相关位置：

| 文件 | 位置 |
|---|---|
| `website/cases.html` | `data-countup` 在 78-88 行附近 |
| `website/assets/js/index.js` | countup 在 674 行附近，案例切换 reset 在 1024 行附近 |

影响：

- 案例页最核心的是“可信数据”，中间值暴露会直接破坏可信度。
- 截图、录屏、客户滚动太快时，都可能截到错误数字。

建议：

1. HTML 初始值写最终值，JS 只做视觉过渡。
2. 动画开始前不要裸露 `0%`。
3. 案例切换时不要把旧值重置为 0 后再显示。
4. 对关键业务数字，宁愿少动效，也不能损害可信度。

### P1-4. 系统演示页轮播 dot 会造成页面纵向跳动

实测：

- `demo.html` 的截图轮播 dot 可点击，active 状态会切换。
- 但点击第 3 个 dot 后，页面被纵向带动，标题区域会跑到固定头部上方，视觉上下文丢失。
- 检测到点击后 `#demo-screens-title` 曾出现在视口上方不可见区域。

相关位置：

| 文件 | 位置 |
|---|---|
| `website/demo.html` | dot 按钮在 126-130 行附近 |
| `website/assets/js/demo.js` | `target.scrollIntoView(...)` 在 18 行附近 |
| `website/assets/css/demo.css` | `.demo-screens-track` 在 95 行附近 |

影响：

- 用户只是想横向切换系统截图，却被页面纵向“拽动”。
- 系统演示页是客户最关心“产品长什么样”的页面，这里必须稳定。

建议：

1. 不用 `target.scrollIntoView()` 控制横向轮播。
2. 改为只控制容器自身的 `track.scrollTo({ left })`。
3. 保证点击 dot 时页面纵向滚动位置不变。

## 6. P2 问题与意见

### P2-1. 新版 7 页首屏结构过于统一

现状：

- 大多数页面都是：深色背景 + 胶囊标签 + 大标题 + 两个按钮 + 右侧抽象图。
- 统一性很好，但看多后会有“同一套 AI 模板批量生成”的味道。

建议：

| 页面 | 建议 |
|---|---|
| 解决方案 | 强化“5 类团队路径图”，而不是只看 GEO 光球 |
| 客户案例 | 首屏直接展示“AI 推荐结果前后对比”或案例证据 |
| 生态资源 | 把联盟网络做成更明确的资源地图 |
| 系统演示 | 让真实后台截图更早、更大、更像产品演示 |
| 关于我们 | 首屏可加入证书/书籍/合作背书的预告 |
| 专属服务 | 保持当前方向，可让 4 阶段流程成为首屏主视觉 |

---

### P2-2. 配色规范与实际页面存在轻微偏差

文档要求：

- 全站主色锁定为电光蓝 `#2F66FF`
- 禁止紫粉渐变

现状：

- `solutions.html` 的 GEO 光球有明显紫色倾向。
- `cases.html` 的数字渐变有紫粉倾向。
- 多个页面的 pill、发光、数字都带紫色。

判断：

- 这些颜色不难看，甚至能提升科技感。
- 但如果严格按 `docs/shared/全站规范.md` 执行，需要统一回电光蓝体系。

建议：

1. 如果客户已经接受当前视觉，可以把规范更新为“电光蓝为主，允许少量蓝紫光效”。
2. 如果规范不可变，则把紫粉压低，避免偏离品牌色。

---

### P2-3. 移动端触控尺寸略小

实测可见元素：

| 元素 | 当前尺寸 | 建议 |
|---|---|---|
| hamburger | `40 x 40` | 至少 `44 x 44` |
| back-to-top | `40 x 40` | 至少 `44 x 44` |
| 移动端顶部预约按钮 | 高度约 `40` | 建议 `44` |

影响：

- 不算严重 bug，但移动端精致度和可点性会下降。

建议：

- 统一把移动端顶部按钮和图标按钮提升到 `44px` 以上。

---

### P2-4. 客户评价仍有 AI 文案味

现状：

- “某知名科技公司”“某上市公司”“某企业服务公司”等匿名表达较多。
- 评价语气接近，像同一套话术。
- 数据很强，但缺少来源解释和业务上下文。

建议：

1. 每条评价加入更具体行业和项目背景。
2. 每条案例按“问题 / 动作 / 结果”展示。
3. 匿名可以保留，但要让行业标签更真实。
4. 核心数据旁边加“统计口径”或“优化周期”。

### P2-5. 大图资源偏重

实测资源：

| 文件 | 体积 |
|---|---|
| `website/assets/system/screen-02.jpg` | 约 4.1MB |
| `website/assets/system/screen-04.jpg` | 约 4.0MB |
| `website/assets/system/screen-01.jpg` | 约 3.7MB |
| `website/assets/system/screen-03.jpg` | 约 3.7MB |
| `website/assets/system/screen-05.jpg` | 约 3.1MB |

## 8. 动效建议

### 8.1 可以加的动效

| 页面 | 动效建议 | 目的 |
|---|---|---|
| 解决方案 | 5 类团队卡片之间用发光路径连接，滚动时路径逐段点亮 | 让“解决方案”不像普通卡片列表 |
| 客户案例 | 案例数据不要直接 countup，可做“优化前 -> 优化后”的对比滑动 | 保留动效，同时提高可信度 |
| 客户案例 | Logo 墙按行业分层轻微浮动，而不是全部同速滚动 | 降低机械模板感 |
| 生态资源 | 资源地图节点 hover 后点亮对应文案卡 | 让生态网络更像真实资源系统 |
| 系统演示 | 截图上加热点 callout，依次标注“监测、引用、风险、推荐率” | 帮客户看懂系统强在哪 |
| 关于我们 | 书籍封面轻微 3D 陈列，证书图滚动进入时做 spotlight | 强化硬背书，不要只像图片卡片 |
| 专属服务 | 4 阶段时间线随滚动填充，当前阶段卡片高亮 | 让服务流程更有交付感 |
| 表单区 | 输入聚焦时边框和标签微亮，提交按钮有轻微 breathing | 提升精致度，但不喧宾夺主 |

### 8.2 不建议加的动效

| 不建议 | 原因 |
|---|---|
| 大面积无业务意义的粒子爆炸 | 容易廉价，且影响性能 |
| 所有卡片同时漂浮 | 会让页面更像 AI 模板 |
| 所有数字都 countup | 关键业务数据会失去可信度 |
| 过多紫粉渐变光效 | 与现有品牌规范冲突 |
| 移动端复杂视差 | 容易卡顿，且信息阅读更重要 |

## 10. 建议方案

### 方案 A：先修硬问题，再增强动效（推荐）

| 项目 | 内容 |
|---|---|
| 实现思路 | 修复溢出、preloader、数字、轮播、旧文件、ICP、大图，再做页面动效增强 |
| 影响范围 | CSS、少量 JS、部分 HTML、交付目录 |
| 成本 | 中 |
| 风险 | 低 |
| 推荐理由 | 最稳，适合正式交付前推进 |

### 方案 B：直接重做全站视觉节奏

| 项目 | 内容 |
|---|---|
| 实现思路 | 每个页面重新定义首屏主视觉和滚动叙事 |
| 影响范围 | 全站结构和视觉 |
| 成本 | 高 |
| 风险 | 中 |
| 推荐理由 | 适合追求更强品牌站效果，但不适合赶交付 |

### 方案 C：只做交付清理

| 项目 | 内容 |
|---|---|
| 实现思路 | 只清旧文件、改 ICP、压缩图片，不改视觉 |
| 影响范围 | 交付目录和资源 |
| 成本 | 低 |
| 风险 | 低 |
| 推荐理由 | 只适合临时演示，不解决高级感和 AI 风问题 |

推荐采用 **方案 A**。
